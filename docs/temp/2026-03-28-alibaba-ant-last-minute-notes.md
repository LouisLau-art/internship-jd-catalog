## 算法题怎么想

### 先判题型

1. 子数组 / 子串 / 最长最短 / 至多至少：
   - 先想滑动窗口
2. 两数之和 / 去重 / 配对 / 出现次数：
   - 先想哈希
3. 有序数组 / 单调性 / 最小最大值：
   - 先想二分
4. 选或不选 / 前 i 个状态 / 最优解：
   - 先想 DP
5. 区间调度 / 最少最多 / 局部最优能推出整体最优：
   - 先想贪心

### 高频模板

- 哈希：
  - `map[value] = index / count`
  - 典型题：去重、两数之和、众数、前缀和计数
- 双指针：
  - 同向：去重、滑窗
  - 反向：有序数组求和、回文、接雨水一类
- 滑窗：
  - 右指针扩张拿到条件
  - 左指针收缩维持条件
  - 适合“最长/最短满足条件子串子数组”
- 二分：
  - 不只是查值，更常见是“二分答案”
  - 前提是答案具有单调性
- 基础 DP：
  - 问自己三件事：
    - 状态是什么
    - 转移从哪来
    - 初始化是什么

## MySQL

### 最容易考的结论

- InnoDB 默认索引是 `B+ 树`，不是红黑树，不是哈希。
- 主键索引的叶子节点直接存整行数据。
- 二级索引叶子节点存 `索引列 + 主键值`，所以常需要回表。
- 覆盖索引能减少回表。
- 联合索引遵循 `最左前缀原则`。
- `select *` 往往更难走覆盖索引。

### B+ 树为什么常考

- 矮胖，磁盘 IO 次数少。
- 叶子节点天然有序，适合范围查询。
- 非叶子节点只存键，更适合做多路搜索。

### 事务与隔离级别

- 事务四特性：`ACID`
- 常见隔离级别：
  - `Read Uncommitted`：可能脏读
  - `Read Committed`：避免脏读，但可能不可重复读
  - `Repeatable Read`：InnoDB 默认
  - `Serializable`：最强，性能通常最差
- InnoDB 的 `Repeatable Read` 主要依赖：
  - `MVCC`
  - 锁机制

### 锁的高频点

- 行锁是加在索引上的，不是“真给这一行贴个锁”。
- 没走索引时，锁范围可能变大。
- 范围查询下，`Next-Key Lock` 用来减少幻读。

### SQL 执行顺序常考口径

1. 连接器
2. 查询缓存
   - 现代版本里这个点常被弱化，但旧题里还会出现
3. 分析器
4. 优化器
5. 执行器
6. 存储引擎

## TCP / 网络

### 三次握手

- 第一次：客户端发 `SYN`
- 第二次：服务端回 `SYN + ACK`
- 第三次：客户端回 `ACK`

### 为什么不是两次

- 需要双方都确认：
  - 自己能发
  - 对方能收
- 也要避免失效的旧连接请求引发问题。

### 四次挥手

- TCP 是全双工，双方都要单独关闭。
- 一方发 `FIN`，另一方先 `ACK`，等自己也发完再发 `FIN`。

### TIME_WAIT 为什么常考

- 保证最后一个 `ACK` 有机会重传。
- 让旧连接中的报文在网络中自然消失，避免污染新连接。

### 可靠传输靠什么

- 序号
- 确认应答
- 超时重传
- 流量控制
- 拥塞控制

### 高频判断题

- TCP 面向连接，UDP 无连接。
- TCP 可靠，UDP 更轻量、时延更低。
- HTTP 是应用层，TCP/UDP 是传输层。
- `ping` 用的是 `ICMP`，不是 TCP。

## Redis

### 高频数据结构

- `String`
- `Hash`
- `List`
- `Set`
- `ZSet`

### 高频场景

- String：缓存、计数器、分布式锁
- Hash：对象字段存储
- List：消息队列、简单列表
- Set：去重、共同好友
- ZSet：排行榜、延时任务

### 过期与淘汰

- 过期不一定立刻删除。
- 常见是：
  - 定期删除
  - 惰性删除
- 内存淘汰常考 `LRU / LFU` 思想。

### 持久化

- `RDB`：快照，恢复快
- `AOF`：追加日志，通常更利于降低数据丢失
- 常见组合是两者一起用

### 缓存三大问题

- 缓存穿透：
  - 查一个本就不存在的数据
  - 常见解法：布隆过滤器、缓存空值
- 缓存击穿：
  - 热点 key 失效瞬间，大量请求打到 DB
  - 常见解法：互斥锁、热点不过期
- 缓存雪崩：
  - 大量 key 同时失效
  - 常见解法：过期时间加随机值、限流、降级

### 一致性高频口径

- 常见策略：先更新 DB，再删缓存。
- 不是先更新缓存再更 DB。
- 更严格时会配合重试、延迟双删、消息队列。
